# 文件治理規範

## 1. 何時必須更新文件

當 PR 變更以下內容時，必須同步更新文件：

- 執行時行為（API／UX／授權／狀態流程）
- 會影響操作或開發流程的相依套件版本
- CI/CD 工作流程、品質關卡或發佈／部署程序
- 安全控制或設定（驗證、CSRF、標頭、機敏資訊處理）
- 貢獻者或維運人員依賴的架構契約
- 重大重構（例如：控制器例外處理機制、Resource 轉換層、測試基礎設施）

若判斷不需要更新文件，PR 說明必須附上簡短理由。

### 重大重構的最低文件要求

符合重大重構條件時，文件必須至少包含：
- 決策理由（為何採此設計）
- 邊界條件（如何擴充、哪些做法禁止）
- 測試策略（如何驗證與避免假陽性/假陰性）
- 維護風險（已知風險與對應緩解）

## 2. ADR 維護規則

### 2.1 何時撰寫 ADR

以下情況必須撰寫新的 ADR：
- 引入新的架構模式或設計模式
- 重大重構或重寫
- 改變模組邊界、職責或相依方向
- 採用新的外部服務或函式庫
- 廢棄或取代現有架構決策

### 2.2 ADR 格式

所有 ADR 必須遵循統一格式，存放於 `docs/decisions/`：

```markdown
# ADR-NNN: 標題

## 狀態

[提案中｜接受｜廢棄｜取代]

## 日期

YYYY-MM-DD

## 背景

為何需要此決策？

## 決策

我們選擇了什麼做法？

## 後果

正面與負面影響分別為何？

## 替代方案

考慮過哪些其他做法？為何不選？
```

### 2.3 ADR 生命週期

- 提案中 → 接受（經團隊審查後）
- 接受 → 廢棄（不再適用）
- 接受 → 取代（由新的 ADR 接替）

## 3. Reader persona 分流檢查

每份文件應標註目標讀者（reader persona），審查時確認：

- [ ] 文件開頭或入口表格明確標示適合的讀者角色
- [ ] 內容深淺與目標讀者相符（管理員文件不應包含原始碼細節）
- [ ] 跨文件連結能引導讀者到對應層級的文件

Reader persona 分類：
| Persona | 適合閱讀 |
|---------|---------|
| 後端開發者 | ADR、領域設計、架構文件、開發者指南 |
| 前端開發者 | 前端架構、API 整合、UI 元件 |
| 內容管理者 | 管理後台操作手冊 |
| 系統管理員 | 系統需求、部署指南、安全設定、故障排除 |
| API 整合者 | API 參考文件 |

## 4. 文件審查檢查清單

審查者應確認：

- [ ] 範圍完整：受影響主題已反映到 canonical 文件
- [ ] 內容正確：指令、路徑、連接埠符合目前儲存庫行為
- [ ] 連結完整：更新後連結都能對到現有檔案
- [ ] 去重一致：主動態文件不存在彼此衝突的重複說明
- [ ] 歸檔整潔：被取代文件已歸檔或移除，並附合理說明
- [ ] 目標讀者明確：文件標示適合的 reader persona
- [ ] （重大重構時）已核對決策理由、邊界條件、測試策略、維護風險四項技術細節

## 5. OpenSpec 與文件同步規則

對每個進行中的 change：

1. proposal／design／tasks 必須標示受影響的 canonical 文件。
2. implementation 完成前，必須先更新對應文件。
3. 最終驗證必須包含受影響文件的連結／路徑檢查。

## 6. Canonical 入口清單

| 路徑 | 說明 |
|------|------|
| `README.md` | 根目錄概覽，5 種 persona 入口 |
| `docs/INDEX.md` | 文件總索引 |
| `docs/decisions/` | 架構決策記錄 (ADR) |
| `docs/domains/` | 領域概述與設計 |
| `docs/architecture/` | 設計文件與模式說明 |
| `docs/frontend/` | 前端架構文件 |
| `docs/api/README.md` | API 文件入口 |
| `docs/guides/` | 各角色指南（admin / developer / frontend / content-creators / deployment） |
| `docs/runbooks/` | 開發與維運 runbook |
| `docs/archive/` | 歷史歸檔文件 |
| `docs/archive/reports/` | 分析報告歸檔 |
