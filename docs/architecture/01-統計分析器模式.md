# 統計分析器模式

## 設計動機

原始設計中，Statistics DTO 同時肩負「資料傳輸」與「商業分析」兩種職責，導致 DTO 檔案膨脹至數百行（`SourceDistributionDTO` 約 200 行，`ContentInsightsDTO` 約 180 行），且分析方法依賴多個內嵌的 `ensure*` 靜態輔助方法。

分析器模式將商業分析邏輯從 DTO 中分離，形成三層架構：

```
DTO（純資料）
  │
  ▼
Analyzer（商業分析）
  │
  ▼
Result（分析結果）
```

## 流程

```
Controller/Service
    │
    ├── 呼叫 Repository 取得原始資料
    ├── 組合為 DTO（僅含純資料 + 簡單衍生欄位）
    ├── 注入對應 Analyzer，傳入 DTO
    │       └── Analyzer.analyze(DTO) → Result
    └── 合併 DTO + Result 回傳
```

### 資料流範例（SourceDistribution）

```
SourceDistributionDTO
  ├── source_data: array        ← 原始資料
  ├── channel_breakdown: array  ← 原始資料
  ├── device_data: array        ← 原始資料
  ├── time_series: array        ← 原始資料
  └── calculated_metrics: array ← 簡單衍生（如總計、比率）
        → 傳入 SourceDistributionAnalyzer.analyze()
              → SourceDistributionResult
                    ├── trafficQualityAnalysis
                    ├── channelPerformanceAnalysis
                    ├── deviceUsagePattern
                    └── trendInsights
```

## 分析器一覽

| 分析器 | 輸入 DTO | 輸出 Result | 分析方法 |
|--------|---------|-------------|----------|
| `StatisticsOverviewAnalyzer` | `StatisticsOverviewDTO` | `StatisticsOverviewResult` | `getActivityLevel()`、`calculateActivityScore()` |
| `SourceDistributionAnalyzer` | `SourceDistributionDTO` | `SourceDistributionResult` | `getTrafficQualityAnalysis()`、`getChannelPerformanceAnalysis()`、`getDeviceUsagePattern()`、`getTrendInsights()` |
| `ContentInsightsAnalyzer` | `ContentInsightsDTO` | `ContentInsightsResult` | `getPerformanceGrade()`、`getContentStrategyRecommendations()`、`getOptimizationInsights()`、`getSeasonalContentStrategy()`、`getReaderBehaviorAnalysis()` |
| `PostStatisticsAnalyzer` | `PostStatisticsDTO` | `PostStatisticsResult` | `getContentQualityMetrics()`、`getEngagementMetrics()`、`getContentAnalysis()`、`calculateQualityScore()`、`calculateAuthorProductivity()` |
| `UserStatisticsAnalyzer` | `UserStatisticsDTO` | `UserStatisticsResult` | `getEngagementAnalysis()`、`getActivityInsights()`、`getPeakActiveHour()`、`getActivityPattern()`、`getWeekendVsWeekdayActivity()` |

## 共享元件

### PerformanceGrade Enum

五個等級對應不同分數區間：

```
PerformanceGrade::fromScore(float $score): self
  A (90-100) → 優異
  B (75-89)  → 良好
  C (50-74)  → 普通
  D (25-49)  → 待改善
  F (0-24)   → 不佳
```

### Recommendation ValueObject

標準化建議格式，含 `type`（improvement/maintenance/alert）、`priority`（high/medium/low）、`message`、`actionable` 屬性。

### ArraySanitizer

集中所有 `ensure*` 靜態清理方法，取代散落在各 DTO 中的重複實作：

- `ensureStringArray()`
- `ensureIntArray()`
- `ensureStringIntArray()`
- `ensureStringNonNegativeIntArray()`
- `ensureNestedStringArray()`

## 測試策略

每個 Analyzer 有獨立單元測試，驗證搬遷前後輸出一致性：

- `SourceDistributionAnalyzerTest`：7+ 測試案例（流量品質分級、管道效能、裝置模式）
- `ContentInsightsAnalyzerTest`：9+ 測試案例（效能等級計算、策略建議、季節性）
- `PostStatisticsAnalyzerTest`：4+ 測試案例（品質指標、互動指標）
- `UserStatisticsAnalyzerTest`：4+ 測試案例（參與度分析、活動模式）
- `StatisticsOverviewAnalyzerTest`：2+ 測試案例（活動等級計算）
