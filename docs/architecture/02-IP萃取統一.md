# IP 萃取統一

## 設計動機

系統中 6 個呼叫者各自實作 IP 萃取邏輯，行為細節不一致（標頭優先順序不同、私有 IP 過濾邏輯不同、Fallback 行為不同），可能導致同一請求在不同端點判斷出不同 IP，影響速率限制與安全審計的正確性。

## NetworkHelper 架構

```
NetworkHelper（單一入口）
  │
  ├── getClientIp(Request, trustedProxies)
  │     └── 原始方法：檢查信任代理 → 依 headerMap 順序取 IP
  │
  ├── getClientIpFromServerParams(serverParams, headerPriority, filterFlags, iterateAllIps, fallback)
  │     └── 新方法：從 $serverParams 依自訂優先順序萃取（精細控制）
  │
  ├── getClientIpWithPrivateCheck(Request, headerPriority, iterateAllIps, fallback)
  │     └── 新方法：信任代理時過濾私有 IP，僅回傳公開 IP
  │
  ├── maskIpAddress(ip): string
  │     └── 從 DeviceInfo 搬遷：IPv4 遮罩末段，IPv6 遮罩半段
  │
  ├── isIpInRanges(ip, ranges): bool
  │     └── 支援：完全比對、CIDR 比對、萬用字元比對
  │
  └── ipInNetwork(ip, cidr): bool
        └── IPv4 CIDR 計算（含 /0 ~ /32）
```

## 6 個呼叫者對應

| 呼叫者 | 使用的方法 | 特殊行為 |
|--------|-----------|----------|
| `JwtAuthorizationMiddleware` | `getClientIpFromServerParams()` + `isIpInRanges()` | 雙重來源（`$serverParams` + `$request->hasHeader()`），過濾私有 IP |
| `RateLimitMiddleware` | `getClientIpWithPrivateCheck()` | 僅信任來自公開 REMOTE_ADDR 的請求 |
| `PostViewRateLimitMiddleware` | `getClientIpFromServerParams(iterateAllIps: true)` | 迭代所有 IP 取第一個公開 IP |
| `StatisticsAdminController` | `getClientIpFromServerParams(fallback: 'unknown')` | 日誌用，無有效 IP 時記錄 `unknown` |
| `CSPReportController` | `getClientIpFromServerParams(fallback: 'unknown')` | 日誌用，同 StatisticsAdminController |
| `DeviceInfo` | `maskIpAddress()` | 委派給 NetworkHelper 靜態方法 |

## 標頭優先順序

```
1. CF-Connecting-IP（Cloudflare）
2. X-Real-IP
3. X-Forwarded-For（取第一個 IP）
4. Client-IP
5. REMOTE_ADDR（fallback）
```

可透過 `$headerPriority` 參數自訂順序。

## 行為比對測試矩陣

為確保每個呼叫者取代前後行為一致，定義行為比對測試計畫：

- **JWT Middleware**：CF header 優先於 X-Forwarded-For；萬用字元 `192.168.*` 比對；CIDR `10.0.0.0/8` 比對
- **Rate Limit Middleware**：私有 REMOTE_ADDR 時信任代理；公開 REMOTE_ADDR 時不信任任何標頭
- **PostView Rate Limit**：多 IP 串列取第一個公開 IP；X-Real-IP 備援
- **Statistics/CSP Controllers**：CF header 優先；無標頭時回退 `'unknown'`
