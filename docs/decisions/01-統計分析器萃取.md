# ADR-1: 統計分析器萃取

**狀態**: 已完成
**日期**: 2026-07-15

## 背景

Statistics DTO 層（`StatisticsOverviewDTO`、`SourceDistributionDTO`、`ContentInsightsDTO`、`PostStatisticsDTO`、`UserStatisticsDTO`）在原始設計中同時肩負兩種職責：

1. **資料傳輸**：攜帶純資料在層之間傳遞
2. **商業分析**：內含 `getActivityLevel()`、`getTrafficQualityAnalysis()`、`getPerformanceGrade()` 等分析方法

這導致 DTO 職責混雜，違反單一職責原則。分析方法依賴大量輔助計算邏輯（如 `ensure*` 靜態方法、`calculateQualityScore`、`calculateContentDiversity`），使 DTO 檔案膨脹至數百行，難以測試與維護。

## 決策

將所有分析方法從 DTO 萃取至 5 個專屬 Analyzer 類別：

- `SourceDistributionAnalyzer`：流量品質、管道效能、裝置使用模式、趨勢洞察
- `ContentInsightsAnalyzer`：效能等級、內容策略建議、優化洞察、季節性策略、讀者行為
- `PostStatisticsAnalyzer`：內容品質指標、互動指標、內容分析
- `UserStatisticsAnalyzer`：參與度分析、活動洞察、尖峰時段、活動模式
- `StatisticsOverviewAnalyzer`：活動等級計算

每個 Analyzer 接收 DTO 輸入，回傳對應的 Result 類別（`SourceDistributionResult`、`ContentInsightsResult` 等）。

同步引入：
- `PerformanceGrade` enum：標準化效能等級
- `Recommendation` ValueObject：標準化建議格式
- `ArraySanitizer` 靜態工具：集中所有 `ensure*` 陣列清理方法

## 結果

**變更檔案**：
- 新增：`Domains/Statistics/Analyzers/`（10 檔案：5 Analyzer + 5 Result）
- 新增：`Domains/Statistics/Enums/PerformanceGrade.php`
- 新增：`Domains/Statistics/ValueObjects/Recommendation.php`
- 新增：`Domains/Statistics/Helpers/ArraySanitizer.php`
- 修改：5 個 DTO — 移除分析方法，改 use `ArraySanitizer`
- 修改：`StatisticsQueryService` — 注入 Analyzer，組合 DTO 與分析結果
- 修改：`config/container.php` — 註冊所有 Analyzer

**對比**：
- **Before**：5 個 DTO 合計約 1500 行，分析方法與資料夾雜
- **After**：5 個 DTO 約 600 行（僅資料 + 簡單衍生），5 個 Analyzer 約 900 行（純商業邏輯）

**測試**：為每個 Analyzer 新增獨立單元測試（約 26 個測試案例），DTO 測試保留純資料存取斷言。

## 替代方案

1. **維持現狀**：DTO 繼續混合職責 — 拒絕原因：違反 SRP，難以測試，DTO 序列化結構包含分析結果不合理。
2. **Analyzer 回傳修改後的 DTO**：讓 Analyzer 直接修改 DTO 屬性 — 拒絕原因：破壞不可變性，副作用難以追蹤。
3. **Service 層直接包含分析邏輯**：將方法移至 `StatisticsQueryService` — 拒絕原因：Service 職責過重，違反關注點分離。

## 取捨

- 呼叫端需多一步注入 Analyzer 並組合結果，但序列化結構更清晰（DTO 僅含原始資料 + 簡單衍生欄位，分析結果獨立區塊）
- 檔案數量增加，但每個類別的職責更明確
