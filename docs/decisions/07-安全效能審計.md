# ADR-7: 安全性與效能審計修復

**狀態**: 已完成
**日期**: 2026-07-15

## 背景

基於安全性審計與效能分析，在 6 個面向發現問題需修正：

### 1. JWT Token 安全
- JWT payload 含 `email` 宣告（非必要個資外洩風險）
- `FirebaseJwtProvider::validateToken()` 使用 `file_put_contents('php://stderr', ...)` 輸出除錯日誌
- JTI 格式非標準 UUID v4
- 缺少角色時效驗證（角色變更後舊 Token 仍可使用的時間窗口）

### 2. Session 安全
- 認證 Cookie 的 `SameSite` 為 `Lax`，面對 CSRF 攻擊保護不足
- 前端 `client.js` 將 Refresh Token 存於 `localStorage`（XSS 可竊取）
- 缺少 CSRF token 補發端點

### 3. 速率限制容錯
- `RateLimitService::checkRateLimit()` 在 Redis 連線失敗時回傳 `allowed: true`（Fails Open），違反安全最佳實踐

### 4. 查詢效能
- `paginate()` 的 N+1 查詢問題：公告列表逐筆查詢標籤
- `UserRepository::paginate()` 使用 `SELECT u.*` 而非明確欄位清單
- `PostCrudRepository::assignTags()` 不必要的 `tagsExist()` 預先檢查

### 5. CSP 與供應鏈安全
- Tailwind CDN 缺少 SRI integrity hash
- CSP 設定未允許 Tailwind CDN 的 `script-src` 與 `style-src`
- `postEditor.js` 直接 `innerHTML` 插入 `tag.name`（XSS 風險）

## 決策

針對每個面向進行修復：

1. **JWT 強化**：建立 `JwtRoleFreshnessValidator` 比對 JWT `iat` 與 `role_updated_at`；移除 payload 中的 `email`；將 `generateJti()` 改為 UUID v4；移除 `php://stderr` 日誌改用條件式 `app_log`
2. **Session 安全**：Cookie `SameSite` 改為 `Strict`；移除前端 `localStorage` Refresh Token 儲存；新增 `GET /csrf-token` 端點
3. **速率限制容錯降級**：`catch (Throwable)` 區塊改回傳 `allowed: false`（Fails Closed）
4. **查詢效能**：新增 `getTagsForPosts(array $postIds)` 批次查詢標籤；`UserRepository` 改為明確欄位清單；移除 `tagsExist()` 預先檢查（讓 DB 外鍵約束處理）
5. **CSP 與供應鏈**：計算並加入 Tailwind CDN SRI hash；更新 CSP 設定允許 `cdn.tailwindcss.com`；`postEditor.js` 改用 `escapeHtml(tag.name)`

## 結果

**變更檔案**：
- 新增：`Domains/Auth/Services/JwtRoleFreshnessValidator.php`
- 修改：`FirebaseJwtProvider` — 移除 `php://stderr` 日誌，JTI 改 UUID v4
- 修改：`AuthenticationService` — 移除 `email` payload
- 修改：`AuthController` — Cookie `SameSite` 改 `Strict`
- 修改：`frontend/js/api/client.js` — 移除 `localStorage` Refresh Token
- 新增：CSRF token 補發路由與 Controller 方法
- 修改：`CsrfMiddleware` — CSRF 補發端點白名單
- 修改：`RateLimitService` — Fails Open → Fails Closed
- 修改：`PostCrudRepository` — 新增 `getTagsForPosts()`，移除 `tagsExist()`
- 修改：`UserRepository` — `SELECT u.*` → 明確欄位清單
- 修改：`SecurityHeaderService` — 更新 CSP 設定
- 修改：`index.html` — Tailwind CDN 加入 SRI hash
- 修改：`frontend/js/pages/admin/postEditor.js` — `innerHTML` 改 `escapeHtml`

**測試**：角色時效驗證測試、Redis 連線失敗測試、批次載入標籤查詢次數驗證、事務回滾測試；`composer check-all` 通過。

## 替代方案

1. **Token 輪替策略**：角色變更時強制登出所有裝置 — 拒絕原因：使用者體驗衝擊過大，角色時效驗證（比對 `iat` 與 `role_updated_at`）更漸進。
2. **使用 Redis 分散式鎖實作速率限制**：— 拒絕原因：對單一 Redis 實例來說鎖的開銷過大，且不能解決 Fails Closed 問題。
3. **資料庫層級 N+1 解法**：JOIN 查詢一次取得標籤 — 拒絕原因：公告與標籤為多對多關係，JOIN 會導致公告資料重複。

## 取捨

- JWT 角色時效驗證增加每次請求的查詢開銷（讀取 `role_updated_at`），但安全性提升顯著
- Rate Limit Fails Closed 可能導致 Redis 故障時短暫誤殺合法請求，但優於繞過速率限制
- 批次標籤查詢增加記憶體使用（需暫存所有關聯），但能將 N 次查詢降為 2 次
