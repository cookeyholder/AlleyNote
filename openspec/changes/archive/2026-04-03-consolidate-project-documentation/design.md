## 背景

近期合併帶來了安全、CI 工作流程、相依套件與測試基礎設施的大量變更。文件內容目前分散在根目錄 README、版本化審查報告、流程說明與 OpenSpec 歷史紀錄中，且存在重疊的安裝與維運指令。貢獻者需要在多份檔案之間來回比對，才能確認正確的本機流程與當前架構假設。

利害關係人：
- 維護者：希望降低文件維護成本
- 貢獻者：需要清楚且可快速上手的指引
- 審查者：需要可重複驗證的「何時必須更新文件」規則

限制條件：
- 在降低日常文件負擔的同時保留必要歷史紀錄
- 儲存庫導覽必須維持簡潔（少量 canonical 入口）
- 不變更執行時邏輯；本次僅處理文件與流程對齊

## 目標與非目標

**目標：**
- 建立具明確唯一來源映射的 canonical 文件資訊架構。
- 整併重複文件並移除過時操作說明。
- 建立治理規則，使後續行為／流程變更必須同步更新文件。
- 透過明確連動，確保 OpenSpec 與儲存庫文件一致。

**非目標：**
- 重構應用程式程式碼或變更 API／執行時行為。
- 將所有歷史報告重寫為新格式。
- 在本次變更導入外部文件平台（wiki／portal）。

## 決策

### 決策 1：Canonical 文件入口
- 選擇：保留少量 canonical 文件（`README.md`、`docs/INDEX.md`、主題 runbook），細節內容收斂到索引化章節。
- 理由：降低導覽成本，消除「哪份文件才是最新」的不確定性。
- 替代方案：
  - 保留所有既有檔案只補連結：拒絕，無法解決內容飄移。
  - 搬到外部文件站：拒絕，超出此次範圍並增加工具負擔。

### 決策 2：主題整併、歷史歸檔
- 選擇：將重複操作內容整併到主題文件（安裝、CI/CD、安全維運），低價值重複文件歸檔或移除。
- 理由：在保留關鍵歷史脈絡下，簡化日常維護。
- 替代方案：
  - 全數硬刪舊文件：拒絕，會失去稽核與歷史追溯價值。
  - 全數保留在主動態文件：拒絕，認知負擔過高。

### 決策 3：把文件更新變成貢獻流程關卡
- 選擇：在貢獻流程加入明確更新觸發與檢查清單，涵蓋行為、相依套件、CI、部署與架構變更。
- 理由：避免後續 PR 再次產生文件落差。
- 替代方案：
  - 僅靠審查者記憶：拒絕，不具一致性。
  - 每個 PR 都強制改文件：拒絕，對純重構或無影響變更過度嚴格。

### 決策 4：OpenSpec 與文件雙向連動
- 選擇：要求每個 active OpenSpec change 標示受影響 canonical 文件，並在 runbook 回連相關能力規格。
- 理由：讓規劃工件與維運文件可持續對齊。
- 替代方案：
  - OpenSpec 與文件各自獨立：拒絕，容易持續分歧。

## 風險與取捨

- [風險] 過度整併導致邊界案例資訊遺失  
  [緩解] 將非日常操作細節保留於附錄或歸檔。
- [風險] 檔案移動後連結失效  
  [緩解] 合併前執行連結與路徑檢查清單。
- [風險] 快速依賴更新 PR 略過文件更新  
  [緩解] 在 PR 模板與審查規則加入必填檢查項。
- [風險] 大型文件 PR 造成審查疲勞  
  [緩解] 以主題切分並提供清楚的搬移／整併矩陣。

## 遷移計畫

1. 盤點現有文件並分類為 Canonical／Merge Target／Archive／Remove。
2. 建立 `docs/INDEX.md`，並將 `README.md` 收斂為 canonical 入口導引。
3. 將重複的安裝／CI／安全說明整併到 canonical runbook。
4. 更新 OpenSpec 參考與能力連結。
5. 驗證連結與命令範例。
6. 移除或歸檔被取代文件。
7. 回滾策略：若遺漏必要資訊，從 git 歷史還原被移除文件。
